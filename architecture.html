架构演进：规划 vs 实现 Architecture Evolution

v1.0 - 端侧 ASR + 本地匹配（已废弃）
手机端做 ASR + AC自动机匹配
App 层承担语音识别 + 敏感词匹配双重职责，逻辑复杂
  • 需要在 App 中集成 Sherpa-ONNX 离线 ASR 引擎
  • 需要维护本地 AC 自动机 + 正则规则引擎
  • ASR 和敏感词匹配分离，无法理解上下文语义
  • 纯关键词匹配，误报率较高（学术讨论 vs 真实危机）
  • App 包体积大，端侧计算资源消耗高
v3.0 - 当前实现（管道式 ASR + LLM）
App 做纯中继，服务器双模式处理
BLE 采集 → iOS 中继 → 服务器 ASR 转写 + LLM 语义分析 → 预警下发，支持本地/云端两种模式
  • App 变薄：只做 BLE 收音 + WebSocket 转发 + 本地通知
  • ASR 模式可切换：FunASR 本地 或 Whisper-compatible 云端 API
  • LLM 支持任意 OpenAI-compatible API（本地/通义/豆包/GPT-4）
  • 预警持久化：SQLite 记录所有预警和转写文本
  • 管理员 Webhook 推送高危预警（Bark/钉钉/飞书/自定义）

大模型 System Prompt 设计 Core Prompt Engineering

关键设计思路

与传统关键词匹配不同，大模型方案的"词库"变成了 System Prompt 中的规则描述。管理员通过修改 Prompt 来调整监控策略，而非维护关键词数据库。这是架构上的根本性简化。

System Prompt 参考模板
你是一个心理咨询会话安全监控助手。你将接收实时的心理咨询对话音频流。

## 你的任务
1. 实时将音频转录为文字，以流式方式输出转录结果
2. 持续分析对话内容，识别以下敏感情况并立即触发预警

## 预警规则

### 一级预警 (HIGH) - 立即触发
触发条件：来访者表达自杀/自伤/伤害他人的真实意图
示例语境：
- "我不想活了" "我想死" "我想结束这一切"
- "我想从楼上跳下去" "我准备了刀片"
- "我想杀了他/她"
- 来访者描述正在遭受虐待、性侵

注意区分：
- 如果是咨询师在做风险评估提问（如"你有没有想过伤害自己"），不触发
- 如果是在讨论过去已解决的经历，降级为二级
- 如果是学术/理论讨论，不触发

### 二级预警 (MEDIUM)
触发条件：
- 来访者表达严重心理危机指标（强烈绝望感、无价值感）
- 咨询师疑似违规执业（保证疗效、劝停药物、不当双重关系）

### 三级预警 (LOW) - 记录不告警
触发条件：需持续关注的风险因素（长期失眠、社交退缩、持续躯体化）

## 输出格式
正常转录时输出纯文字。检测到需预警内容时，在转录中插入 JSON 标记：

[ALERT:{"level":"HIGH","trigger":"来访者说'我不想活了'",
"context":"来访者在描述近期压力时突然表达自杀意念，语气低沉，非学术讨论",
"suggestion":"建议立即进行自杀风险评估"}]

## 重要原则
- 宁可漏报不可误报：只在有较高把握时触发一级预警
- 理解语境：同样的词在不同上下文中含义完全不同
- 尊重专业性：不要对咨询师的正常专业操作触发误报

预警分级体系 (大模型语义驱动) AI-Driven Alert Classification

预警等级 分类说明 大模型判断依据（非简单关键词） 响应方式 响应时效
HIGH 自杀/自伤/伤害意图 来访者以第一人称表达当前/近期的自杀意念、自伤行为或伤害他人计划，语境为真实表达而非学术讨论或咨询师评估提问 系统通知（锁屏）+ App 弹窗 + Webhook 推管理员 < 3s
HIGH 虐待/性侵披露 来访者描述正在遭受或近期遭受身体虐待、性侵害，且非回顾已解决事件 系统通知（锁屏）+ App 弹窗 + Webhook 推管理员 < 3s
MEDIUM 严重心理危机 来访者表达强烈绝望感、无价值感、放弃感，但未明确表达自杀意图 App 通知 + Webhook 推管理员 < 5s
MEDIUM 违规执业嫌疑 咨询师承诺治愈效果、建议停药、暗示不当关系、违反保密原则 App 内展示（不推通知） < 10s
LOW 需关注的风险因素 来访者提到长期失眠、社交退缩、持续躯体化症状、回避行为加重等 静默记录 + 会话结束后汇总 会话后

语义驱动预警的核心优势

简单关键词匹配中 "想死" 这个词只要出现就触发预警。而 LLM 语义分析方案能理解：

-- "我真的想死了，每天都在想从窗户跳下去" -> 触发 HIGH（真实自杀意念）

-- "你有没有想过死亡？" -> 不触发（咨询师正常评估提问）

-- "我去年想过死，但现在好多了" -> 触发 LOW（历史事件，需记录但无需告警）

-- "这篇关于自杀预防的论文提到..." -> 不触发（学术讨论）

这就是管道式 ASR + LLM 方案的核心价值 -- 语义级理解，而非字面匹配。

风险与应对 Risks & Mitigations

⚠
网络依赖
服务器端强依赖网络连接
  • 风险：断网时完全无法工作，服务器不可达则无转写和预警
  • 应对：心理咨询室通常有稳定 WiFi，断网概率很低
  • 兜底：App 端可保留一个极轻量的本地关键词匹配作为断网降级方案
  • 监控：App 实时检测网络状态，断网时提示咨询师
🔒
隐私与合规
音频上传至云端的隐私风险
  • 风险：音频流经网络，需保证传输加密和服务器安全
  • 应对：使用自建服务器（Spark2），音频不落盘，仅内存处理
  • 加密：WebSocket TLS 加密传输，音频不在云端持久化
  • 合规：来访者知情同意书中明确说明数据处理方式
✓
场景优势不变
心理咨询室天然适合此方案
  • 安静环境：大模型语音识别在安静环境下准确率极高
  • 稳定网络：咨询室有固定 WiFi，网络质量有保障
  • 语义场景固定：心理咨询对话模式相对固定，Prompt 容易调优
  • 对延迟不敏感：1-2s 延迟在心理咨询场景完全可接受
💡
Prompt 调优是关键
预警质量取决于 Prompt 设计
  • 核心工作：Prompt 的质量直接决定误报率和漏报率
  • 建议：与北师大心理专家合作设计 Prompt 规则
  • 迭代：收集真实场景数据持续优化，支持 A/B 测试
  • 管理：管理后台提供 Prompt 版本管理和效果对比

v3.0 实现总结 -- 管道式 ASR + LLM

架构：XIAO nRF52840 Sense（PDM 采集）→ BLE Nordic UART Service → iPhone App（Swift/SwiftUI 纯中继）→ WebSocket → 服务器（FunASR 转写 + LLM 语义分析）→ 预警 JSON 下发。

最大优势：LLM 天然具备上下文语义理解能力，能区分"真实自杀意念"和"学术讨论"，这是传统关键词匹配无法做到的。误报率大幅降低，预警质量本质性提升。

灵活性：ASR 支持本地 FunASR 或任意 Whisper-compatible API（env 切换）；LLM 支持本地模型或任意 OpenAI-compatible API（通义/豆包/GPT-4 等改 env 即可）；Webhook 支持 Bark/钉钉/飞书/自定义 POST。
